Personal Secure Web Gateway

### Background ###

Modern end users frequently browse the internet without visibility into the security posture of visited websites, downloaded content, or credential usage. Common risks include unsafe browsing, malicious links, credential reuse across services, and lack of behavioral insight into personal network activity. Traditional enterprise security controls (secure gateways, SIEM monitoring, Zero Trust verification) are typically unavailable to individuals.
This project addresses these risks by designing a personal secure web gateway that intermediates all browser traffic, performs automated reputation checks, securely manages credentials locally, and provides continuous visibility and verification for a single user environment.

This project designs and implements a personal secure web gateway firewall intended for a single end user within a VMware laboratory environment. The system acts as an intermediary between the user and the internet, ensuring that all web traffic passes through a controlled security gateway before reaching external services.

Unlike manual scanning tools, the platform performs automated inspection, monitoring, and decision-making transparently in the background. The goal is to demonstrate realistic cybersecurity architecture concepts while remaining implementable by a single student developer.

The platform combines multiple defensive concepts into a modular personal security ecosystem:
	• Secure proxy firewall controlling outbound traffic
	• Automated URL and simulated file reputation scanning
	• Local encrypted password manager
	• Traffic metadata collection and behavioral analysis
	• Personal SIEM-style logging and visualization
	• Zero Trust verification principles
	
Deployment consists of two VMware virtual machines:
	1. User VM – browser and Angular dashboard
	2. Gateway VM – NestJS security platform and database

### Method ###

System Architecture Overview

The proxy operates at the HTTP application layer (OSI Layer 7). It analyzes and mediates web requests at the protocol level rather than performing packet-level inspection or acting as a network firewall. System focuses on application-layer security controls such as request validation, reputation checks, authentication enforcement, and logging, while excluding lower-layer packet inspection or kernel-level filtering.
All traffic is mediated through a central proxy. All security validation occurs before traffic is forwarded:
Browser → Gateway Proxy → Internet

Backend Architecture (NestJS Monolith with Modules)

Modules:
	• ProxyModule – traffic interception and forwarding
	• AuthModule – authentication and Zero Trust checks
	• ThreatModule – external reputation scanning
	• VaultModule – password manager
	• SIEMModule – logging and analytics
Internal communication uses an event-driven model via NestJS EventEmitter.

A modular monolith architecture was intentionally selected instead of microservices due to security focus over infrastructure complexity. NestJS modules provide strong boundaries similar to microservices while remaining within a single deployable application. Modules can later be extracted into independent services for distributed deployment.

Request Processing Pipeline

Failure Handling Policy
Security decisions must explicitly define behavior when external dependencies fail. The gateway follows a fail-safe but usable model:
	– VirusTotal API timeout or unavailable → Request allowed with "Unverified" warning logged and risk score increased.
	– Scanning service error response → Default allow with security event logged for audit visibility.
	– Internal proxy error → Request blocked and custom error page returned.
	– Database unavailable → Sensitive operations (authentication, vault access) denied; proxy continues operating in limited logging mode.
	– Authentication verification failure → Request blocked and session invalidated.
All failure events are recorded in security_events for later analysis and displayed in the dashboard

Zero Trust Enforcement

No request is implicitly trusted regardless of network location. This system applies Zero Trust principle by requiring verification for every interaction, independent of whether traffic originates from the local network, trusted VM, or previously authenticated session.
Each request verifies:
	• Valid JWT session
	• Active session record
	• Device fingerprint (IP + User-Agent)
	• Recent authentication timestamp
Sensitive actions require re-authentication:
If now - last_auth_at > 20 minutes: require password conformation

Cryptography Model

Threat Model Assumptions
The cryptographic design is based on the assumptions what the attacker may do: gain access to the database storage or observe encrypted traffic between components. Attacker does not have access to runtime memory of the gateway process while the user is authenticated and does not know the user's master password.
Security goals under these assumptions:
	– Database compromise must not reveal stored credentials.
	– Captured traffic must remain confidential.
	– Encryption keys exist only in RAM memory and are re-derived after authentication.
	
Authentication
Password hashing: Argon2id
Session tokens: JWT in HttpOnly Secure cookies

Key derivation:
Master Password + User Salt
        ↓
Argon2 KDF
        ↓
Vault Encryption Key (memory only)

Encryption algorithm: AES-256-GCM
Stored values:
- Ciphertext
- IV
- Authentication tag

Database Schema

Performance and Indexing Strategy
Because dashboard analytics and real-time monitoring rely heavily on querying log data, database indexing is required to maintain performance. The traffic_logs table is expected to be the largest dataset and therefore includes optimized indexes:
	• Index on timestamp → enables fast time-range queries for dashboards.
	• Composite index on (user_id, timestamp) → supports per-user activity timelines.
	• Index on verdict → accelerates filtering by allowed/blocked traffic.
	• Optional index on risk_score → improves analytics and anomaly views.

Event-Based WebSocket Design
WebSocket communication follows an event-driven messaging model to simplify frontend subscription handling and RxJS stream composition. Each message contains an explicit event type allowing the dashboard to react selectively.
Event types:
	• TRAFFIC_LOG — emitted when a proxied request is processed
	• SECURITY_EVENT — emitted for warnings, failures, or detected threats
	• SESSION_EVENT — authentication or re-verification updates
	• SYSTEM_STATUS — gateway health or dependency status

Angular services map events into RxJS streams:
	- traffic$ → filters TRAFFIC_LOG
	- securityEvents$ → filters SECURITY_EVENT
	- systemStatus$ → filters SYSTEM_STATUS
This approach enables scalable real-time UI updates while keeping frontend logic modular and aligned with the backend event architecture. RxJS Observables power real-time updates.
